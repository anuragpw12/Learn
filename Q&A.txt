52.SOLID Principle -

-> The acronym of the 5 design principles helps developers to create software that is more maintainable, flexible, and scalable.

	1.(S)Single Responsibility Principle (SRP): 
		- A class should have one reason to change (one responsibility)
		- Bad Eg. -
			public class Invoice
			{
				public void CalculateTotal()
				public void PrintInvoice()
				public void SaveToDB()
			}
		- Good Eg. - 
			public class Invoice
			{
				public void CalculateTotal(){}
			}
			public class InvoicePrinter
			{
				public void PrintInvoice(){}
			}
		- SRP → CreateOrderHandler (does only order creation)
		
	2.(O)Open/Closed Principle (OCP): 
		- Classes should be open for extenstion but close for modification.
		- Use abstraction and polymorphism instead of modifying existing code.
		- Bad Eg. -
			public class AreaCalculator
			{
				public double Area(object shape)
				{
					if(Shape == Circle)
						 return Math.PI * Radius * Radius;
					else if(Shape == Rectangle)
						return w*h;
				}
			}
		- Good Eg. - 
			public abstract class Shape
			{
				public abstract double Area();
			}
			
			public class Circle:Shape
			{
				public double Radius { get; set; }
				public override double Area()
				{
					return Math.PI * Radius * Radius;
				}
			}
			
			public class Rectangle:Shape
			{
				public double w { get; set; }
				public double h { get; set; }
				public override double Area()
				{
					return w*h;
				}
			}
		
		- OCP → IPaymentService (extend via new implementations)

	3.(L)Liskov Substitution Principle (LSP):
		- Subclasses should be replacable for their base classes without breaking behavior.
		- Subclasses should be replaceable for their base classes means that you should be able to use a 
		  child class object in place of a parent class object without changing how the program behaves or breaking its logic.
		- Bad Eg.
			public class Bird
			{
				public virtual void Fly();
			}
			public class Ostrich:Bird
			{
				Ostrich cant fly
			}
		- Good Eg.
			public abstract class Bird(){}
			
			public class FliyingBird:Bird
			{
				public void Fly(){}
			}
			
			public class Ostrich:Bird
			{
				public void Run(){}
			}
			
		- LSP → RazorpayService : IPaymentService 
		- (CashPayment , OnlinePayment )

	4.(I)Interface Segregation Principle (ISP):
		- No client should be forced to depend on methods it doesn’t use.
		- Break large interface into smaller,specifed one.
		- Bad Eg.
			public interface IMachine
			{
				void Print();
				void Scan();
				void Fax();
			}
			public class OldPrinter:IMachine
			{
				Print();
				Scan(); - Not implemented
				Fax(); - Not implemented
			}
			
		- Good Eg.
			public interface IPrinter
			{
				void Print();
			}
			public interface IScanner
			{
				void Scan();
			}
		- ISP → IUserCommandService, IUserQueryService
		
	5.(D)Dependency Inversion Principle (DIP):
		- High level modules should not depend on low level modules, both should depend on abstraction.
		- Use interface + Dependency Injection
		- Bad Eg.
			public class EmailService
			{
				public void SendEmail(string message)
			}
			public class Notification
			{
				EmailService _es = new EmailService();
				_es.SendEmail("test mail");	
			}
		- Good Eg.
			public interface IMessageService {
				void Send(string message);
			}
			public class EmailService : IMessageService {
				public void Send(string message) {
					// Email sending logic
				}
			}
			public class Notification {
				private readonly IMessageService _service;

				// Inject the dependency via constructor
				public Notification(IMessageService service) {
					_service = service;
				}

				public void Notify(string message) {
					_service.Send(message);
				}
			}
		
		- DIP → IOrderRepository (used by Application, implemented in Infrastructure)

	“SOLID appears in Clean Architecture through handlers, interfaces, and dependency injection.”


53. Covariance & Contravariance
	They describe how generic types behave with inheritance.
	The core problem
		Given:
			class Animal { }
			class Dog : Animal { }
		Can this work?
			List<Dog> dogs = new List<Dog>();
			List<Animal> animals = dogs; // ❌ not allowed
		
		Why?
			Because generics are invariant by default.
	
	Covariance (out) - Dog → Animal (upcast) Covariance allows child to parent.
	Contravariance (in) - Animal → Dog (reverse direction) Contravariance allows parent to child.

.NET Runtime Flow
C# code → IL (EXE or DLL) → EXE process starts → CLR loads → JIT compiles IL → native code runs → GC manages memory → done

1. You write code in C#
	Human-readable .cs source files.
	
2. C# compiler compiles to IL
	The C# compiler (csc.exe) produces:
		IL code (Intermediate Language)
		Metadata
		Output → EXE or DLL
	✔ EXE = has Main()
	✔ DLL = library (no entry point)
	IL is not machine code yet.
	
3. A worker process starts (ALWAYS an EXE)
	This is very important:
		If you run a console app, your EXE runs.
		If you run MVC / Web API, IIS (w3wp.exe) is the EXE.
		If .NET Core, dotnet.exe is the EXE.
	✔ Every .NET application executes inside an EXE process.
	
4. CLR loads inside the process
	CLR = Common Language Runtime (the .NET Runtime)
	It initializes:
		JIT compiler
		Garbage Collector (GC)
		Security
		Thread Pool
		Type System
		Assembly Loader
	CLR is now ready to run IL.

5. CLR loads assemblies (your EXE + DLLs)
	CLR reads:
		IL code
		Metadata
		Manifest
		✔ Loading = preparing
		✔ NOT executing yet

6. JIT compiles IL → native machine code

	When the program calls a method for the first time:
		CLR sends the IL to JIT
		JIT converts IL → native CPU instructions
		Native code is stored in memory
		Next time → runs native code directly (no JIT)
	JIT runs only once per method unless the app restarts.

7. Native code executes + GC runs
	Once compiled, the native code runs.
	During execution:
		Objects are allocated on the managed heap
		GC automatically frees unused memory
		ThreadPool runs background tasks
		Exceptions, type safety, and security handled by CLR
	✔ CLR manages the whole runtime environment

8. Program finishes
	When the EXE process ends:
		CLR shuts down
		All DLLs unload
		GC cleans up
		OS ends the process

When I was working in WinForms, I had a progress bar with marquee style.
Initially I called the report generation method normally, and the progress bar was not moving because the UI thread was busy doing the heavy work. Since WinForms has only one UI thread, it could not repaint the screen.
Then I used async/await and ran the long operation asynchronously. The work moved to a background thread and the UI thread became free, so the progress bar animation started moving smoothly.
From that I understood async/await doesn’t make execution faster — it just prevents blocking and keeps the application responsive.

7. Thread Pool (Workers Factory) -

		- Slow
		- Wasteful if created repeatedly

	So instead:
		- Threads are created once
		- Reused again and again
		- Reuse workers instead of hiring new workers every time

	- How it works (real meaning)
		- One task starts
		- CPU/thread stays busy or waiting
		- Nothing else runs on that thread
		- When the task finishes → next line executes

2. Asynchronous Programming -
	- Asynchronous programming means a task is started, but the thread does NOT wait for it to finish.
	  The thread is free to do other work, and the result is handled later when the task completes.
	- How it works (real meaning)
		- Task starts
		- Thread releases control
		- Thread can handle other requests
		- When task finishes → result is picked up
	- eg . You order food in a restaurant:

	- Parallel programming means executing multiple tasks at the same time using multiple CPU cores to improve performance.

		- You assign 25 parcels to each → work finishes faster
	- eg. RabbitMQ

15. ref and out and  in
	ref and out are parameter modifiers
	ref and out both pass arguments by reference 	
	
	- in is used to pass a variable by reference but as read-only for performance optimization.
	
	- ref
		The variable must be initialized before being passed
		The method can read and modify the value
	- out
		The variable does not need to be initialized
		The method must assign a value before returning
		If out is not assigned a value inside the method before it returns, the C# compiler throws a compile-time error
	
	- Pass by Value
		A copy of the variable is sent to the method
		Changes inside the method do not affect the original value
		Pass by value is useful when you don’t want a method to change your original data
	- Pass by Reference
		The actual variable (its memory address) is sent to the method
		Changes inside the method do affect the original value
	
16. Stack and Heap -
	Stack - 
		- A fast, small, LIFO (Last In, First Out) memory area.
		- Used to store method calls and local variables.
		- Method call frames
		- Local variables (value types)
		- Method parameters (value types)
		- References (addresses) to heap objects
		- Memory is allocated and freed automatically
		- Very fast
		- Size is limited
		- Data lifetime = method scope
		
	Heap -
		- A large, slower, shared memory area.
		- Used to store objects created with new.
		- Reference type objects (class, string, array)
		- Object data (fields)
		- Boxed value types
		- Memory allocated dynamically
		- Managed by Garbage Collector (GC)
		- Objects live beyond method scope
		- Slower than stack
		- Static variables are stored on the HEAP, inside the AppDomain / process memory, once per type.
		- Static class = container of static data on heap
		
	class Person
	{
		public string Name;
	}

	void CreatePerson()
	{
		Person p = new Person();
		p.Name = "Anurag";
	}

	Memory behavior
	- p (reference) → stack
	- Person object → heap
	- Stack stores only the address
	- Heap stores the actual object
	

17. Monolithic Architecture -

	In a monolithic application:

		Business logic

18. Microservices Architecture -
	Microservices Architecture is a design approach where an application is built as a collection of small,

19. Tier Architecture
	Tier Architecture is a design pattern where an application is divided into logical or physical layers (tiers), 
	and each tier has a specific responsibility.
	
	2-Tier Architecture - 
		- In 2-Tier architecture, the application is divided into Client tier and Database tier, 
		  where the client communicates directly with the database.
		- Client (UI + Business Logic)
			 ↓
		  Database
		- Example 
			Desktop apps (WinForms, WPF)
			Client directly using SQL Server
			Small desktop apps
	
	3-Tier Architecture - 
		- 3-Tier architecture separates the application into Presentation, Business, and Data tiers.
		- Presentation Tier (UI)
				↓
		  Business Tier (Logic)
				↓
		  Data Tier (Database)
		- Example
			Presentation → ASP.NET MVC / Web API
			Business → Services
			Data → Repository / EF / SQL
			Most web applications
			
	N-Tier Architecture - 
		- N-Tier architecture extends 3-Tier by adding more specialized layers for better scalability and flexibility.
		-   UI
			 ↓
			API Layer
			 ↓
			Business Layer
			 ↓
			Domain Layer
			 ↓
			Data Access Layer
			 ↓
			Database
		- Example
			Enterprise & large systems
		
		
20. Layered Architecture
	- Layered Architecture is a design approach where an application is organized into logical layers, 
	  each with a specific responsibility, and each layer communicates only with the layer directly below it.
		Presentation Layer
				↓
		Application / Business Layer
				↓
		Domain Layer
				↓
		Data Access / Infrastructure Layer
		
21. Layered vs Tier Architecture
	- Layered 
		- code organization
		- Meaning Logical separation
		- Layers Code-level
		- Deployment Same or separate
		- Focus Maintainability
		
	- Tier
		- deployment structure
		- Meaning Physical deployment
		- Layers Server-level
		- Deployment Often separate servers
		- Focus Scalability
		
22. Layered vs Clean Architecture
	- Layered
		- Dependency direction Top → Bottom
		- Business isolation - Partial
		- Framework independence - Low–Medium
		- Business layer often depends on EF, SQL, frameworks
	
		public class OrderService
		{
			private readonly AppDbContext _db;

			public OrderService(AppDbContext db)
			{
				_db = db;
			}
		}
		
		❌ Business depends on EF
		❌ Hard to test without DB

	- Clean Architecture
		- Dependency direction Outside → Inside
		- Business isolation Strong
		- Framework independence High
		- Dependencies always point inward
		
		public class CreateOrderUseCase
		{
			private readonly IOrderRepository _repo;

			public CreateOrderUseCase(IOrderRepository repo)
			{
				_repo = repo;
			}
		}

		✔ Business depends on abstraction
		✔ EF lives outside (Infrastructure)
		
		Testing difference (very practical)
			Layered
				Need DB or heavy mocks
				Slower tests
			Clean
				Pure unit tests
				No DB needed
				Very fast
		
		Layered architecture can become Clean if:
			You apply Dependency Inversion
			Business layer depends only on interfaces
			Infrastructure implements those interfaces
		
		Layered architecture separates code by responsibility, while Clean Architecture enforces dependency inversion to protect business rules—even
		if both look visually similar.

24.Cryptography-
	- Cryptography is the science of protecting information so that:
		- Only authorized parties can read it
		- Data cannot be altered without detection
		- Sender identity can be verified
		
	- It provides:
		- Confidentiality
		- Integrity
		- Authentication
		- Non-repudiation
		
	- Cryptography has three major categories:
		| Category       | Purpose                   | Examples             |
		| -------------- | ------------------------- | -------------------- |
		| Encryption | Protect data (reversible) | AES, RSA             |
		| Hashing  | Verify data (one-way)     | MD5, SHA-256, bcrypt |
		| Signing    | Prove authenticity        | RSA signatures       |

	- MD5 vs bcrypt vs SHA-256 (clear comparison)
		| Feature        | MD5         | SHA-256           | bcrypt |
		| -------------- | ----------- | ----------------- | ------ |
		| Type           | Hash        | Hash              | Hash   |
		| Speed          | Very Fast ❌ | Fast ❌            | Slow ✅ |
		| Salt           | ❌ No        | ❌ No (by default) | ✅ Yes  |
		| Password safe  | ❌ No        | ❌ No              | ✅ Yes  |
		| Collision safe | ❌ No        | ✅ Yes             | ✅ Yes  |

		
	- Symmetric Encryption
		- Symmetric Encryption uses ONE single key for:
			- Encryption
			- Decryption
		- Plain Text + Secret Key → Encrypted Data Encrypted Data + Same Secret Key → Plain Text
			
		- Characteristics
			- Very fast
			- Best for large data
			- Key sharing is risky

		- Example
			- AES (Advanced Encryption Standard)
		- Usage
			- Encrypting files
			- Encrypting database columns
			- Encrypting API payloads (after key exchange)		
					
			
	- 1. AES (Advanced Encryption Standard)
		- A symmetric encryption algorithm
		- Uses fixed key sizes:
			- 128-bit
			- 192-bit
			- 256-bit
						
		- Why AES is used
			- Extremely fast
			- Highly secure
			- Industry standard

		- Common Modes
			- AES-CBC
			- AES-GCM (recommended)
						
			
	- Asymmetric Encryption
		- Asymmetric Encryption uses TWO keys:
			- Public Key	Encrypt / Verify
			- Private Key	Decrypt / Sign
		
		- Characteristics
			❌ Slower than symmetric
			✅ Very secure
			✅ No key sharing problem
		- Example
			- RSA
					
	- RSA(Rivest–Shamir–Adleman)
	
	- An asymmetric encryption algorithm
	- Uses public & private key pair
	- RSA is mainly used for:
		- Encrypting small data
		- Key exchange
		- Digital signatures
	- RSA is NOT used to encrypt large payloads
	
	- Hashing - 
		- Hashing is a one-way process:
			- Cannot be reversed
			- No decryption possible
		- Characteristics
			- One-way only
			- Used for verification, not recovery
		- Example
			- bcrypt

	- bcrypt (Password Hashing)
		- A password hashing algorithm
		- Includes:
			- Salt
			- Multiple rounds
			- Slow by design	

		- Why bcrypt is used
			- Protects against brute-force attacks
			- Even if DB is stolen, passwords are safe
		- bcrypt is NOT encryption
		- bcrypt cannot be decrypted	
		- Why NOT create your own hashing algorithm?
			- Custom logic is reversible or predictable
			- No protection against brute-force attacks
				- Attackers use:
					- GPUs
					- Billions of guesses per second
			- Built-in salt (automatically)
			- Industry-proven & audited
			
		- What is a Brute-Force Attack?
			- A brute-force attack is when an attacker tries every possible combination of 
			  characters until the correct password or key is found.


	- RSA + AES (Hybrid Encryption)
		- Why Hybrid Encryption?
			| Problem                              | Solution         |
			| ------------------------------------ | ---------------- |
			| AES is fast but key sharing is risky | Use RSA          |
			| RSA is secure but slow               | Use only for key |
			| Best of both worlds                  | RSA + AES        |

		- Hybrid Encryption – Encryption Flow
			Step 1: Generate AES Session Key
				A random AES key is generated per request
			Step 2: Encrypt AES Key using RSA Public Key
				Only the private key holder can decrypt it
			Step 3: Encrypt Payload using AES
				JSON / Request body encrypted with AES key	

		- Client
			 ├─ AES encrypt JSON payload
			 ├─ RSA encrypt AES key (public key)
			 └─ Send both to server

		- Hybrid Encryption – Decryption Flow
			Step 1: Decrypt AES Key
				Server uses RSA private key
			Step 2: Decrypt Payload
				AES key decrypts JSON payload
		
		- Server
			 ├─ RSA decrypt AES key (private key)
			 └─ AES decrypt payload → Plain Text


25. JWT Authentication

	- JWT (JSON Web Token) is a secure way to send information between client and server in JSON format.
	- It is mainly used for:
		- Authentication
		- Authorization 
	A JWT has 3 parts separated by dots:
		- header.payload.signature
	What each part means
		Header: algorithm info (e.g., HS256)
		Payload: claims/data (userId, role, exp…)
		Signature: created using a secret/private key → prevents tampering
	= JWT payload is Base64 encoded, not encrypted.
	= Access Token - (short life, e.g., 15 min)
	= Refresh token (longer life)
	= Client sends token in header: Authorization: Bearer <JWT>

=>Bearer means: “whoever is holding (bearing) this token is allowed.”
=>standard format of the Authorization header

	- JWT Flow
		- Question: Who issues the token?
		👉 Server / API
			- User logs in using username & password
			- Server validates credentials
			- Server generates JWT
			- Token is sent back to client
			
	- JWT Structure
		- JWT consists of 3 parts, separated by dots (.):
		- xxxxx.yyyyy.zzzzz

		- 1. Header
			Contains:
			 - Algorithm (e.g. HS256)
			 - Token type (JWT)

		- 2. Payload
			Contains:
				- Claims
				- User data

			Example:
				- sub (user id)
				- name
				- role
				- exp

		- 3. Signature
			- Generated using:
				Header
				Payload
				Secret Key
			- Ensures integrity & authenticity
	
	
	- Audience
		Question: Who is the token for?
		👉 Client / Frontend
			- Token is issued for a specific application
			- Prevents misuse by other apps
	- Claims
	- Used for: User info
		Examples:
		- UserId
		- Username
		- Role
		- Email
					
	- Expiry
		- Token expiry time
			- Token has a fixed lifetime
			- Example: 15 min / 1 hour
			- Prevents infinite usage

	- Signature
		- Purpose: Security
			- Token is signed using a secret key
			- Ensures token is not tampered

	- Invalid Token
		- If token is:
			- Expired
			- Modified
			- Invalid signature

	👉 Server returns 401 Unauthorized
	 
	- Signing Method
		- HS256 (HMAC SHA-256) is commonly used
		- Uses secret key

	- Token Storage (Client Side)
		JWT is usually stored in:
			- Local Storage
			- Session Storage
			- HTTP-only Cookies (recommended)

	- JWT Validation (Server Side)
		Server validates:
			- Token format
			- Signature
			- Expiry time
			- Issuer
			- Audience
			
		If valid → request allowed
		If invalid → 401 Unauthorized

	- Invalid Token Response
		401 Unauthorized
	
	- JWT Usage in API
		- Token is sent in HTTP Header
		- Authorization: Bearer <JWT_TOKEN>
	
	- Important JWT Security Points
		- Never store secret key in frontend
		- Always use HTTPS
		- Use short-lived access tokens
		- Use refresh tokens if required
	
	Flow = 
		1. User logs in
			Server sends:
				- Access token (response body)
				- Refresh token (HttpOnly cookie)
		2. Client uses access token
			- Sent in Authorization header
			- Valid for 15 minutes
			- No refresh call during this time
		3. Access token expires
			- API returns 401 Unauthorized
		4. Client calls /refresh
			- Browser automatically sends cookie
			- No JS access to refresh token
		5. Server does ALL of this:
			- Reads refresh token from cookie
			- Hashes & checks DB
			- Verifies not revoked
			- GENERATES NEW refresh token
			- Revokes old refresh token
			Issues:
				New access token
				New refresh token (new cookie)

		✔ This is refresh token rotation

Kestrel is a lightweight, high-performance web server used by .NET to host web applications.

In old .NET Framework:
	IIS was mandatory.
	App could not run without IIS.

In .NET Core / .NET:
Kestrel allows:
	Self-hosting
	Linux hosting
	Docker containers
	Cloud-native apps

	Opens a port (5000, 7000, etc)

		HTTP/HTTPS
		Routing requests into middleware
		Sending responses back

The == operator must be overloaded separately if we want value comparison.

Identity = same memory
Equality = same meaning/data

Eager Loading using Include loads related data in a single query.

Eager loading is generally preferred in Web APIs to avoid N+1 query issues.

# ✅ LINQ – Element Operators (One Line Each)

**First()** – Returns the first matching item and throws an exception if none exist.

**FirstOrDefault()** – Returns the first matching item or default (null) if none exist, without throwing an exception.

**Single()** – Returns exactly one matching item and throws an exception if none or more than one exist.

**SingleOrDefault()** – Returns one matching item or null if none exist, but throws an exception if more than one exist.

# ✅ Dapper – Core Concepts (One Line Each)

**Dapper** – A lightweight ORM that maps raw SQL query results directly to C# objects with high performance.

**Query()** – Executes a query and returns multiple records mapped to objects.

**QueryFirst()** – Returns the first row from the result set and throws if no rows exist.

**QueryFirstOrDefault()** – Returns the first row or null if none exist, without throwing an exception.

**QuerySingle()** – Expects exactly one record and throws if none or multiple records exist.

**QuerySingleOrDefault()** – Returns one record or null if none exist, but throws if multiple records are found.

**Execute()** – Executes INSERT, UPDATE, or DELETE commands and returns the number of affected rows.

**ExecuteScalar()** – Returns a single value (first column of the first row), commonly used for aggregate queries.

**QueryMultiple()** – Executes a query that returns multiple result sets and reads them using Read<T>().

Data Structure - 
-----------------------------------------------------------------------------------------------------------------------------------------	
1.What ?
	- A Data Structure is a way of organizing and storing data in memory so that it can be used efficiently.
	- Data = information (numbers, text, objects)
	- Structure = how that data is arranged
	
	- Two Types - 
		- 1.Primitive
			- int, float/double, char, bool, pointer
			
		- 2.Non-Primitive (derived / user-defined)
		
			1. Linear Data Structures (elements in a sequence)
				- Array (static, contiguous memory)
					- Array is a collection of similar type values stored one after another in memory and accessed using index numbers.
					- Static Array = int[] arr = new int[5];
					- Dynamic Array (List) = List<int> list = new List<int>();
 
				- Linked List (dynamic, non-contiguous)
					- A linked list is a linear data structure in which elements are stored in separate nodes, 
					  and each node contains data and a reference (pointer) to the next node in memory.
					- Singly Linked List - Each node has: Data Pointer to next node 
					- Doubly Linked List - Each node:Data Next Previous
					- Circular Linked List - Last node points to first
					
				- Stack (LIFO)
					- A stack is a linear data structure that follows the principle of LIFO (Last In First Out),
 					  where insertion and deletion are performed only from one end called the top.
					- Push (Insert) stack.Push(10);
					- Pop (Delete) stack.Pop();
					- Peek (Top element) stack.Peek(); Peek = just look To see the top element, not delete.
					
				- Queue / Deque (FIFO / both-ends)
					- A queue is a linear data structure that follows the principle of FIFO (First In First Out), 
					  where insertion happens at the rear and deletion happens from the front.
					- q.Enqueue(10);
					- Deque (Double Ended Queue) 
						- A deque is a linear data structure in which insertion and deletion can be performed 
						  from both the front and the rear.
						- deque.AddFirst(10);
						- deque.AddLast(20);
					- Types of Queue
						Simple Queue
						Circular Queue
						Priority Queue
						Deque
					- Real systems - RabbitMQ,Kafka,Redis(deque)
					
				- String (often treated as linear)
					

			2. Non-Linear Data Structures (hierarchical / network)
				- Trees
					- A tree is a non-linear data structure that stores elements in a hierarchical manner, 
					  where each node can have zero or more child nodes, and there is exactly one root node.
					- Binary Tree 
						- Each node has at most 2 children.
						    10
						   /  \
						  5   20

					- Binary Search Tree(BST)
						- A binary search tree is a binary tree where: 
						- Left subtree contains smaller values
						- Right subtree contains greater values
							        50
								   /  \
								 30    70

													
					- Heap
						- A heap is a complete binary tree that follows heap property.
						- Types of Heap
						- Min Heap Parent ≤ Children
								      5
									 / \
								   10  20
						- Max Heap Parent ≥ Children
							  20
							 / \
						   10   5
						   
					- B-Tree / B+ Tree
						- A multi-level tree where each node can have more than two children.
						- Used in:
							Databases
							File systems
							     [10 | 20]
							   /    |     \
							 [5]  [15]   [25]

					- Trie (Prefix Tree)
						- A tree used to store strings where each node represents a character.
						- Used in:
							Auto-complete
							Spell check
						
					- AVL 
						- An AVL tree is a self-balancing binary search tree in which the 
						  difference in height between the left and right subtrees of any node is at most 1.
						  
				- Graphs
					- Directed/Undirected, Weighted/Unweighted
					- A graph is a non-linear data structure consisting of a 
					  set of vertices (nodes) and a set of edges (connections) that connect pairs of vertices.
					- Types of Graphs
						1. Undirected Graph = Edges have no direction. A ---- B eg. Facebook friends
							Means:
								A is connected to B
								B is connected to A
						2. Directed Graph (Digraph) = Edges have direction. A ---> B eg. Instagram follow
							Means:
								A influences B
								But B does not influence A 
								
					- Graph Traversal
						Traversal = visiting all nodes.		
							1. BFS (Breadth First Search)
								Level by level.
								Flow:
									Visit all neighbors first
									Then neighbors of neighbors
									
							2. DFS (Depth First Search)
								Go deep first.
								Flow:
									Go as far as possible
									Then backtrack
						
				- Hashing
					- Hashing is a technique used to convert a given key into a 
					  fixed-size value (called a hash) using a hash function, so that data can be stored and retrieved efficiently.
					- Hash Function - A hash function is a function that takes an input (key) and returns an integer (hash value).
						Key: "India"
						Hash: 42
					- Meaning: Store "India" at index 42.
					- Hash Table - A hash table is a data structure that stores key-value pairs using hashing.
					
			
		- Searching
			- Searching is the process of finding a specific element in a collection of data.
			1. Linear Search - Linear search checks each element one by one until the target is found.
			2. Binary Search- Binary search repeatedly divides the data into halves to find the element.
				Flow
					Check middle
					If smaller → go left
					If bigger → go right
					Repeat
			3. BST search
			- eg .
				        50
					   /  \
					 30    70
					/ \    / \
				  20  40  60  80
				  
				1. Inorder Traversal (L → Root → R)
					- Left → Root → Right
					- 20 → 30 → 40 → 50 → 60 → 70 → 80

				2. Preorder Traversal (Root → L → R)
					- Root first, then children.
					- 50 → 30 → 20 → 40 → 70 → 60 → 80
					
				3. Postorder Traversal (L → R → Root)
					- Children first, then parent.
					- 20 → 40 → 30 → 60 → 80 → 70 → 50
				
		- Sorting
			- Sorting is the process of arranging data in a specific order, usually ascending or descending.
			- Types of Sorting
				1. Bubble Sort 
					- Repeatedly swaps adjacent(next) elements if they are in wrong order.
					- Largest elements "bubble" to the end.
					- Time: O(n²)
				2. Selection Sort 
					- Selects the smallest element and places it at correct position.
					- Time: O(n²)
				3. Insertion Sort
					- Builds sorted list one element at a time.
					- Like arranging playing cards.
					- Time: O(n²) worst
				4. Merge Sort
					- Divides data into halves, sorts them, then merges.
					- Divide and conquer.
					- Time: O(n log n)
				5. Quick Sort
					- Selects a pivot and partitions data around it.
					- Average: O(n log n)
				6. Heap Sort
					- Uses heap structure to extract min/max repeatedly.
					- Time: O(n log n)
			

2.Time Complexity

	- Time Complexity describes how the execution time grows as the input size increases.
	- It’s expressed using Big-O notation.

	| Big-O      | Name         | Meaning           |
	| ---------- | ------------ | ----------------- |
	| O(1)       | Constant     | Same time always  |
	| O(log n)   | Logarithmic  | Divide & conquer  |
	| O(n)       | Linear       | Loop once         |
	| O(n log n) | Linearithmic | Efficient sorting |
	| O(n²)      | Quadratic    | Nested loops      |
	| O(2ⁿ)      | Exponential  | Recursion heavy   |
	| O(n!)      | Factorial    | Permutations      |

	// O(n)
	for(int i = 0; i < n; i++)
	{
		Console.WriteLine(i);
	}
	// O(n²)
	for(int i = 0; i < n; i++)
	{
		for(int j = 0; j < n; j++)
		{
			Console.WriteLine(i + j);
		}
	}

3.Space Complexity (SC)
	- Space Complexity measures extra memory used by the algorithm.
	Includes:
		Variables
		Arrays
		Recursion stack
		Data structures
		
	// O(1) space
	int sum = 0;
	for(int i = 0; i < n; i++)
	{
		sum += i;
	}
	
	// O(n) space
	int[] arr = new int[n];

	| Method          | Time     |
	| --------------- | -------- |
	| From first page | O(n)     |
	| Binary search   | O(log n) |
	
	| Case         | Meaning              |
	| ------------ | -------------------- |
	| Best Case    | Minimum time         |
	| Average Case | Typical              |
	| Worst Case   | Maximum time (Big-O) |

	Example: Searching in array
	Best: first element
	Worst: last element → O(n)

	- Can be redeclared
	- Can be reassigned
	- Gets hoisted (initialized as undefined

	Eg - 
		function testVar() {
			if (true) {
				var x = 10;
			}
			console.log(x); // 10 (accessible outside block)
		}
		testVar();
		
		Redeclare allowed:
			var a = 5;
			var a = 20;
			console.log(a); // 20

		Hoisting:
			console.log(b); // undefined
			var b = 10;

	Bug - 
		for (var i = 0; i < 3; i++) {
			setTimeout(() => {
				console.log(i);
			}, 1000);
		}
		// Output: 3 3 3


2. let (Modern – block scoped)
	- Block scoped { }
	- Cannot be redeclared in same scope
	- Can be reassigned
	- Hoisted but in Temporal Dead Zone (TDZ)
	
	- Eg. 
		function testLet() {
			if (true) {
				let y = 10;
			}
			console.log(y); // ❌ Error: y is not defined
		}
		
		Reassign allowed:
			let a = 5;
			a = 20;
			console.log(a); // 20

		TDZ:
			console.log(c); // ❌ Error
			let c = 10;

		
		Bug in var fixed in let -
			for (let i = 0; i < 3; i++) {
				setTimeout(() => {
					console.log(i);
				}, 1000);
			}
			// Output: 0 1 2
		
		- let is block scoped, var is function scoped.
		
		
3. const (Constant reference)

	- Block scoped
	- Cannot be redeclared
	- Cannot be reassigned
	- Must be initialized
	- For objects/arrays → reference is constant, not values		
	
	- Eg.
		const x = 10;
		x = 20; // ❌ Error
	
		With object:
			const user = { name: "Anurag" };
			user.name = "Rahul";   // ✅ Allowed
			user = {};             // ❌ Error

		With array:
			const arr = [1,2,3];
			arr.push(4);  // ✅
			arr = [];     // ❌

Use const by default, let if value changes, avoid var.

4.What is Hoisting
	- Hoisting is JavaScript’s behavior of moving declarations to the top of scope before execution.
	- console.log(a); // undefined
	  var a = 10;

5.Difference between == and ===

	| ==               | ===               |
	| ---------------- | ----------------- |
	| Loose comparison | Strict            |
	| Type conversion  | No conversion     |
	| "5" == 5 → true  | "5" === 5 → false |

	- Always use === in real projects.
	
	
6.What is Closure?

	- A closure is when a function remembers variables from its parent scope even after the parent function has finished.
	- Closures keep variables in memory.
	- Scope = room where variable lives
	- Closure = lock that keeps the room from being destroyed

	- Used in: data privacy, caching, debounce.
		function counter() {
		  let count = 0;
		  return function () {
			return ++count;
		  }
		}
	- eg.Closure for Data Hiding (Encapsulation)
		function bankAccount(balance) {
			return {
				deposit(amount) {
					balance += amount;
				},
				getBalance() {
					return balance;
				}
			};
		}

		const acc = bankAccount(1000);
		acc.deposit(500);
		console.log(acc.getBalance()); // 1500

11.Shallow vs Deep Copy

	- Shallow copy → both objects point to the same nested objects
	- Deep copy → completely independent memory

14. What is Debounce?
	- Debounce delays execution until user stops action.
		Used in:
			Search box
			Resize

15.What is Throttle?
	Throttle limits execution to once per time interval.

	Used in:
		Button clicks
		Infinite scroll

	Without throttle:This runs hundreds of times per second.
	With throttle (every 1 second):No matter how fast you scroll It logs once per second


	| Throttle                    | Debounce                  |
	| --------------------------- | ------------------------- |
	| Executes at fixed intervals | Executes after user stops |
	| Good for continuous events  | Good for final action     |
	| Eg: scroll position         | Eg: search box            |

	| Parameters              | (start, end)   | (start, deleteCount, items...) |

Mediator + CQRS

1) HTTP Request → Controller
2) Controller creates Command object
3) Controller sends Command to Mediator
4) Mediator finds Validator and runs it
5) If valid → Mediator calls Handler

RabbitMQ
RabbitMQ is a message broker — it transfers messages between services asynchronously.

Producer — sends a message (job request) to RabbitMQ.

Queue — temporary storage where messages wait for processing.

Consumer (Worker) — background service that processes messages.

Message — data payload (your ReportJobMessage).

Broker — RabbitMQ server managing queues and routing.

Exchange & Routing

Exchange — routes messages from producer to correct queue.

Routing Key — address label used by exchange to decide destination queue.

Default Direct Exchange — built-in exchange where routingKey = queueName.

Your Implementation Mapping

Controller — Producer trigger
RabbitMqPublisher — sends message
report_jobs queue — stores report requests
ReportJobConsumer — worker processes report
DB status tracking — Pending → InProgress → Done/Failed
ACK after processing — ensures no duplicate execution


| Exchange       | Behavior         | Real Example               |
| -------------- | ---------------- | -------------------------- |
| Default/Direct | 1 → 1            | Report job                 |
| Fanout         | 1 → many         | Send notifications         |
| Topic          | Pattern routing  | Event-driven microservices |
| Headers        | Metadata routing | Rare                       |

Redis
What Redis is

Redis is an in-memory database used mainly as a cache to make reads extremely fast.

Redis — an in-memory key-value data store used mainly for caching and fast data access.

In-Memory Storage — data is stored in RAM so reads are extremely fast.

Distributed Cache — shared cache accessible by multiple application servers.

Source of Truth — database remains permanent storage, Redis is only a performance layer.

Expiration & Memory Control

TTL (Time To Live) — automatic key expiry after specified time.

Absolute Expiration — key removed after fixed duration.

Key Design

Key Naming Convention — structured keys like tracking:{awbNo} for targeted invalidation.

NUnit – Unit Testing Framework for .NET

NUnit is a unit testing framework for .NET used to test individual methods, classes, or components in isolation.
It ensures your code works correctly, prevents regressions, and improves maintainability.

1. Test Fixture

A class marked with `[TestFixture]` indicating it contains unit tests.

[TestFixture]
public class CalculatorTests
{
}

2. Test Methods

Each method that should be executed as a test is marked with [Test].

[Test]
public void Add_ReturnsCorrectSum()
{
}

## 3. Assertions

Assertions verify expected outcomes.

Assert.AreEqual(expected, actual);
Assert.IsTrue(condition);
Assert.IsFalse(condition);
Assert.Throws<TException>(() => method());
Assert.That(value, Is.EqualTo(expected));

4. Setup and Teardown

Used to initialize objects before each test and clean up after each test.

[TestFixture]
public class CalculatorTests
{
    private Calculator _calc;

    [SetUp]
    public void Init()
    {
        _calc = new Calculator();
    }

    [TearDown]
    public void Cleanup()
    {
        _calc = null;
    }
}

5. Testing a Calculator Class

Class Under Test

public class Calculator
{
    public int Add(int a, int b) => a + b;

    public int Divide(int a, int b)
    {
        if (b == 0)
            throw new DivideByZeroException();

        return a / b;
    }
}

Test Class

[TestFixture]
public class CalculatorTests
{
    private Calculator _calc;

    [SetUp]
    public void Setup()
    {
        _calc = new Calculator();
    }

    [Test]
    public void Add_WhenCalled_ReturnsCorrectResult()
    {
        int result = _calc.Add(3, 5);
        Assert.AreEqual(8, result);
    }

    [Test]
    public void Divide_ByZero_ThrowsException()
    {
        Assert.Throws<DivideByZeroException>(() => _calc.Divide(10, 0));
    }
}

6. Parameterized Tests

Run the same test with multiple inputs.

[TestCase(2, 3, 5)]
[TestCase(10, 20, 30)]
[TestCase(-1, -1, -2)]
public void Add_WithMultipleInputs_ReturnsExpected(int a, int b, int expected)
{
    var calc = new Calculator();
    Assert.AreEqual(expected, calc.Add(a, b));
}

7. Mocking Dependencies

NUnit does **NOT** provide mocking.
Use libraries like **Moq** or **NSubstitute**.

 Example Scenario

Service depends on a repository.

public interface IUserRepository
{
    User GetUser(int id);
}

public class UserService
{
    private readonly IUserRepository _repo;

    public UserService(IUserRepository repo)
    {
        _repo = repo;
    }

    public string GetUserName(int id)
    {
        var user = _repo.GetUser(id);
        return user?.Name;
    }
}

Test with Moq

[TestFixture]
public class UserServiceTests
{
    [Test]
    public void GetUserName_ReturnsCorrectName()
    {
        // Arrange
        var mockRepo = new Mock<IUserRepository>();
        mockRepo.Setup(r => r.GetUser(1))
                .Returns(new User { Name = "John" });

        var service = new UserService(mockRepo.Object);

        // Act
        var result = service.GetUserName(1);

        // Assert
        Assert.That(result, Is.EqualTo("John"));
    }
}

Deep Understanding

Mocking:

* Removes external dependencies (DB, API, filesystem)
* Makes tests fast and repeatable
* Ensures isolation
* Tests only business logic, not infrastructure

Difference between CPU-bound and IO-bound operations?

CPU-bound tasks depend on CPU processing power, like calculations or encryption.
IO-bound tasks depend on external resources like database, file system, or network.
For IO-bound tasks, we use async/await to improve scalability. For CPU-bound tasks, we use multi-threading or parallel programming.
